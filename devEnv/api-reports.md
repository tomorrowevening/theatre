# API Reports

Each public package has a single-file API surface snapshot at `<package>/etc/<package>.api.md`. These files are generated by [API Extractor](https://api-extractor.com/) and committed to git. They serve as the canonical declaration of the public API surface.

## Packages with API reports

- `packages/dataverse/etc/theatre-dataverse.api.md`
- `packages/react/etc/theatre-react.api.md`
- `packages/theatric/etc/theatric.api.md`
- `theatre/core/etc/theatre-core.api.md`
- `theatre/studio/etc/theatre-studio.api.md`

## How it works

API Extractor analyzes each package's `dist/index.d.ts` (the rolled-up declaration file) and produces a single Markdown file listing every exported symbol — types, interfaces, classes, functions, variables — with their full signatures.

There are two modes:

1. **CI mode** (`yarn run validate:api`) — runs `api-extractor run` without `--local`. Generates a temporary report and **compares it against the committed snapshot**. If they differ, the build fails with a warning that says "You have changed the API signature for this project." Run this in CI to catch unintended public API changes.

2. **Local update mode** (`yarn run update:api-snapshots`) — runs `api-extractor run --local`. Regenerates the committed snapshots from the current source. Use this when you have intentionally changed the public API and want to update the snapshots.

## Workflow

When you change the public API of a package:

```bash
# 1. Make your code changes
# 2. Build the affected package(s)
yarn workspace @tomorrowevening/theatre-dataverse run build

# 3. Update the snapshots
yarn run update:api-snapshots

# 4. Review the diff in `etc/*.api.md` files
git diff -- '**/etc/*.api.md'

# 5. Commit the snapshot changes alongside your source changes
git add packages/dataverse/etc/theatre-dataverse.api.md
git commit
```

If you run `yarn run validate:api` after making API changes without updating snapshots, you will get a warning like:

```
Warning: You have changed the API signature for this project.
Please copy the file ".temp/api-reports/<package>.api.md" to "etc/<package>.api.md",
or perform a local build (which does this automatically).
```

In CI, this warning fails the build (exit code 1).

## Why commit generated artifacts?

API report files are committed to git for the same reason Jest snapshots are: they act as a regression test for the public API surface. A diff in `etc/*.api.md` during code review immediately surfaces any unintended changes to public types, signatures, or exports — including:

- Accidental removal of an exported symbol (breaking change)
- A return type narrowing or widening
- A new public function added without intent
- A type parameter rename that affects consumers
- A `@public` symbol becoming `@internal` or vice versa

Reviewers can read the diff and ask "is this change intentional?" — the same way they'd review a snapshot test diff.

## Why a single `.api.md` file per package?

Earlier, the dataverse package used `typedoc` + `typedoc-plugin-markdown` to generate a tree of per-class/interface markdown files under `packages/dataverse/api/`. This was inconsistent (only one of five packages had it), and the format produced large diffs for small changes (one file per symbol). API Extractor's single-file report is:

- **Consistent across all packages** — every package follows the same convention
- **Diff-friendly** — adding one export creates a few-line diff in one file
- **Self-validating** — the same tool that generates the file also enforces it in CI
- **No extra dependencies** — `@microsoft/api-extractor` was already in the project; `typedoc` and `typedoc-plugin-markdown` could be removed entirely
